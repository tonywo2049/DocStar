# Methodology 兼容预设（Agent 协作工程方法论 → DocStar）

给按「Agent 协作工程方法论」模板库写作的项目一套即拷即用的 DocStar conventions：语料一字不改，点亮分类、符号索引与规格链覆盖检查（唯有意非链接分桶例外——须写作端给条目打前缀词，见可改槽位）。`corpus/` 是一个忠实于模板形制的迷你实例项目（ROADMAP + M1 需求/设计/任务/决议/交接；其上下游条目的 `外部：`/`泛指：` 前缀为 DocStar 侧标记、非模板形制），被测试锁定，兼作可跑样例。

## 用法

```bash
# 1. 看效果（本 fixture 即样例）
python3 docstar.py check --corpus fixtures/methodology/corpus
# 2. 用到你的项目：把预设拷进语料根，自动发现
mkdir -p <你的语料根>/.docstar/conventions
cp fixtures/methodology/corpus/.docstar/conventions/conventions.json <你的语料根>/.docstar/conventions/
```

## 预设声明了什么

| 块 | 内容 | 对应方法论出处 |
|---|---|---|
| `nature_source` | 元信息块 `类型` 字段 → 性质映射（表见下）；可选 `normalize` 归一括号附注值 | 文档元信息块模板：五字段无 `性质`，性质语义住在 `类型` |
| `def_forms` | `### R<n>`（需求）/`- **R<n>-AC<m>**`（验收标准）/`- **NFR<k>**`/`\| **T<n>.<m>** \|`（任务卡行）/`## §<N>`（决议）/`**Q<n> ——`（设计问题） | 需求文档/任务文档/决议记录/设计文档模板 |
| `doc_id_kinds` | 同上各 ID 的全文提及索引 + `M<n>` 里程碑 | ROADMAP/分配账本的表格引用形 |
| `form_headers.ledger` | `\| 日期 \| 版本 \| 变更 \|`（四列变更记录，兼容默认三列） | 需求/设计模板 §变更记录 |
| `form_headers.glossary_section` | 增列「前提与公共定义」为术语节 | 需求文档 §A |
| `required_edges` | 「AC 须有任务卡声明」（规格链：追踪矩阵未覆盖的 AC=未实现） | 方法论 §2.B 规格链可机检承诺 |
| `task_columns` | 任务卡表列名：`规格锚`/`前置`/`失败先行测试`/`状态` | 任务文档模板 任务卡清单表头 |
| `edges.nonlink_prefixes` | 有意非链接前缀词表：`外部：`（链根/仓外产物/会话类）、`泛指：`（集合性下游） | 元信息块模板 上游/下游 纪律——链根与「下游=下一会话/各执行文档」类条目天然无链接（前缀词本身为预设自造，模板库无此形） |

### 类型 → 性质映射表

| → 规范（进判定域） | → 记述（图有、不进门禁） |
|---|---|
| 结论型 · 目标/目标文档 · 需求文档 · 设计文档 · 接口契约/契约 · ROADMAP · 决议记录 · 核对单 · 模板 · 任务文档 | 过程型 · 调研 · 复盘 · 交接 handoff · 跨轨登记册 · 分配账本 |

依据方法论 §2.A 类型学：被下游依赖、变更须传导的=规范；滚动状态/过程记录=记述。任务文档虽含滚动台账，但任务卡表的规格锚声明正被规格链检查依赖（『被下游依赖』判据），而记述源的声明不进门禁——故定性 规范（2026-07-17 裁决）。显式 `性质` 键恒最高（拼错或空值=unknown，不被映射掩盖）；未入表的 `类型` 值=unknown（不猜；带全角括号附注的复合值先经下段归一，归一后仍未入表才 unknown）。

括号附注归一（`nature_source.normalize: "bracket-base"`）：原值精确命中优先；未命中且形如「基值（附注）」（尾部全角括号）时剥附注取基值再查一次，仍未命中=unknown（不猜，语义不变）；活例=语料 `决议记录（例会）` 剥注后落 `决议记录` 命中→规范。

## 可改槽位（按你项目实况调）

- **任务卡号**：预设 `T<n>.<m>`（跨轨登记册 `<x.1>` 占位同形）。方法论模板未钦定卡号语法——你的项目若用别的形（如 `TA1.2`），同步改 `def_forms.任务` 与 `doc_id_kinds` 任务行。
- **任务表列名**：`task_columns` 四列名按你的表头改（识别判据=首列 `#` 且 spec 列名在表头；键缺席回落引擎内置 `spec 锚`/`前置`/`红先测试`/`状态`）。
- **命名空间锚**：`namespaces.req_doc`/`task_doc_stem` 等按你的实际文件名改（本预设写的是 fixture 的文件名）。
- **映射表**：`nature_source.map` 是本预设的裁量，你的项目对某类型另有定性，改表即可。注意耦合：规格链覆盖检查只认 规范 源的任务声明（记述=不进门禁）——把 任务文档 改回 记述 会让「AC 须有任务卡声明」恒报未覆盖。
- **括号归一**：`nature_source.normalize: "bracket-base"` 是可选增强，不要此行为（如你的 `类型` 值从不带括号附注、或想让带附注的值一律 unknown）删 `normalize` 键即回精确匹配（EG-26 原语义，逐字节不变）。
- **有意非链接词表**：`edges.nonlink_prefixes` 按你的标记习惯改（如把 `[]` 字面量也收进词表，可豁免 `下游: []` 这类写法）；分桶生效前提=写作端给条目逐条打前缀（本语料已示范 3 条）；词表缺席则回落全计——纯文字上下游条目不论是否天然无链接，一律照旧全报 `fm_无链接条目`。

## 在方法论模板库自身上跑

模板库本身也是语料（整仓含顶层 README 共 18 篇）：零配置即出完整上下游链；套本预设后未分类 18→2（仅剩两个无 frontmatter 的 README，如实上报）。实测抓到过模板库一处真缺陷：信任面登记表→核对单-关账前 单向边（元信息块模板明文「单向指针两头一致是纪律」）。

## 已知边界（诚实清单）

- **表格与有序列表条目不走零配置兜底**——方法论的登记表几乎全是表格（任务卡/映射/门禁/线状态），须由本预设 `def_forms` 显式声明；`1. **X**` 有序列表条目（ROADMAP TODO、登记册决策点）现无抽取通道，暂为纯文本。
- **`v` 前缀版本号**：变更记录用 `v0/v1`，引擎修订解析认 `r` 前缀——CHK-3 传导检查保持休眠（预设不声明 `revision_target_kinds`，不装能）。
- **决议 `§N` 不设 ID kind**——`决议记录 §1` 这类引用恰是 DocStar 原生节引用通道（`id "决议记录 §1"` 直接可查），双通道会重复索引；决议实体由 `## §N` 标题 def_form 抽取。
- **「验证声明须覆盖 AC」未激活**——方法论无测试文档模板，声明即假红风暴；你的项目有测试落盘形后自行加规则。
